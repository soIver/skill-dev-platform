# AGENTS.md

## Назначение
- Этот репозиторий представляет собой веб-платформу для оценки и развития навыков ИТ-специалистов.
- Перед изменением моделей, связей, логики сохранения или поведения функций, зависящих от хранимых данных, прочитайте `backend/src/models.py`.

## Источник истины
- `backend/src/models.py` является источником истины для целевой модели данных.
- Если текущий код противоречит `backend/src/models.py`, не сохраняйте код молча. Укажите на несоответствие и устраните его осознанно.

## Требование к обновлению документации

При внесении **глобальных изменений**, затрагивающих архитектуру, бизнес-логику, модели данных, API-контракты или основные рабочие процессы, вы должны обновить этот файл.

### Что считается глобальным изменением:
- Изменения схемы базы данных или моделей (`models.py`)
- Добавление или удаление основных сущностей
- Изменения конечных точек API или форматов запросов/ответов
- Изменения бизнес-логики, затрагивающие несколько модулей
- Изменения конфигурации окружения или развертывания
- Изменения ключевых алгоритмов или валидаций

### Что НЕ требует обновления:
- Незначительные исправления ошибок
- Рефакторинг без изменения поведения
- Изменения пользовательского интерфейса (UI/UX) на фронтенде
- Добавление вспомогательных утилит

## Карта репозитория
- `backend/` содержит бэкенд на FastAPI и настройку Celery-воркера.
- `backend/src/main.py` связывает FastAPI-приложение, инициализацию при запуске, CORS и ограничение частоты запросов; конечные точки должны располагаться в доменных роутерах.
- `backend/src/models.py` определяет модели SQLAlchemy.
- `backend/src/auth/` содержит маршруты аутентификации, схемы, защиту запросов и логику токенов/сессий.
  - Для администраторов доступен `GET /auth/owners`, который возвращает список пользователей с ролью `curator` или `admin` для фильтров по владельцу на фронтенде.
  - Конечные точки подтверждения почты при регистрации находятся в `POST /auth/email-confirmation/request`, `GET /auth/email-confirmation/verify` и `POST /auth/email-confirmation/complete`.
  - Принятие приглашения на роль куратора существующим пользователем выполняется через `POST /auth/curator-invitation/confirm`; успешное подтверждение инвалидирует все сессии пользователя.
  - `POST /auth/register` используется только для завершения регистрации через GitHub OAuth; стандартная регистрация не должна обходить email-confirmation.
  - Конечные точки смены пароля через письмо находятся в `POST /auth/password-change/request`, `GET /auth/password-change/verify` и `POST /auth/password-change/confirm`.
  - Конечные точки смены почты через письмо находятся в `POST /auth/email-change/request`, `GET /auth/email-change/verify`, `POST /auth/email-change/request-confirmation` и `POST /auth/email-change/confirm`.
  - Редактирование имени пользователя использует `GET /auth/username-availability` для проверки доступности и `PATCH /auth/username` для сохранения с перевыпуском cookie-токенов.
  - Авторизованный пользователь при переходе на `/auth/login` или `/auth/registration` должен перенаправляться на `/profile`.
- `backend/src/github/` содержит OAuth GitHub и логику подключенных профилей.
- `backend/src/mail/` содержит сервис отправки HTML-писем, шаблоны писем и Redis-коды подтверждения; API-роутеры для подтверждающих действий располагаются в доменных модулях.
- `backend/src/management/` содержит административные маршруты и сервис управления кураторами: список кураторов/администраторов, приглашения по email, отмену приглашений и отзыв роли куратора.
- `backend/src/classifier/` содержит маршруты, схемы и сервис управления классификатором компетенций: профессиональными стандартами, ОТФ и ТФ.
- `backend/src/vacancies/` содержит маршруты поиска вакансий HeadHunter, схемы, нормализацию фильтров и маппинг ответов.
- `backend/src/tasks/` содержит API-маршруты заданий, схемы, функциональность поиска и логику сохранения.
- `backend/src/tests/` содержит API-маршруты, схемы и сервис прохождения тестов; активная попытка хранится в Redis, итоговый результат сохраняется в `test_attempts`.
- `backend/src/utils/` содержит общие утилиты бэкенда: криптографию, настройку базы данных, логирование.
- Профессиональные стандарты серии 06 хранятся в `prof_standards`, группы трудовых функций — в `ps_functions_groups`, трудовые функции — в `ps_functions`.
- `frontend/` содержит клиент на React + TypeScript, собранный с помощью Vite.
- `frontend/src/components/` содержит переиспользуемые UI-компоненты:
  - `ActionMenu.tsx` — выпадающее меню действий для таблицы
  - `AuthBootstrap.tsx` — проверка/инициализация авторизации пользователя при старте приложения
  - `AutocompleteSearch.tsx` — поле ввода с автодополнением и выпадающим списком
  - `BentoSearch.tsx` — гибкий компонент поиска и выбора элементов с поддержкой debounce
  - `EditorConfirmModal.tsx` — модальное окно подтверждения отмены изменений в редакторе
  - `ErrorBoundary.tsx` — предохранитель для перехвата ошибок в компонентах React
  - `FieldRequirements.tsx` — визуальный список требований к полям ввода
  - `HorNavBar.tsx` — горизонтальная навигационная панель
  - `IconButton.tsx` — кнопка с иконкой
  - `InfoModal.tsx` — модальное информационное окно
  - `PaginatedTable.tsx` — таблица с поддержкой пагинации данных
  - `Pagination.tsx` — универсальный компонент пагинации (стрелки и текст «Страница х из y»)
  - `RangeSlider.tsx` — переиспользуемый двусторонний слайдер диапазона значений
  - `RequireAuth.tsx` — обертка для защиты маршрутов, проверяющая авторизацию
  - `RootRedirect.tsx` — перенаправление пользователя с корневого маршрута в зависимости от его статуса
  - `TabTracker.tsx` — отслеживание вкладок/разделов
  - `ToastProvider.tsx` / `ToastViewport.tsx` — провайдер и область отображения всплывающих уведомлений (Toast)
  - `TestCard.tsx` — карточка опубликованных тестов по навыку с переключением уровней
  - `VacancyCard.tsx` — настраиваемая карточка отображения вакансии с HeadHunter
  - `VerNavBar.tsx` — вертикальное боковое меню навигации
- `frontend/src/routes.tsx` является основной картой маршрутов.
- `frontend/src/auth.ts` содержит клиентские запросы аутентификации/сессий и логику обновления токенов.
- `frontend/src/hooks/useStore.ts` содержит общее состояние фронтенда через Zustand.
- `frontend/src/hooks/useTestsStore.ts` содержит состояние поиска публичного банка тестов через Zustand.
- `docker-compose.yml` запускает PostgreSQL и Redis для локальной разработки.

## Примечания для будущих агентов
- Если добавляются новые подсистемы, обновите карту репозитория и рабочие правила здесь.
- Модель `Task` включает обязательные поля: `title` (длина 4-48, уникальное) и обязательное `description`. API выполняет поиск по обоим полям, когда предоставлено ключевое слово.
  - Поиск контентных сущностей в разделе управления поддерживает фильтр по владельцу через параметр `author_id`; куратор может фильтровать только по собственному `user_id`, администратор — по любому куратору.
  - Фильтрация заданий по `skill_level_ids` при поиске работает по логике ИЛИ: задание подходит, если содержит хотя бы один из выбранных уровней навыков.
- Для тестов (модель `Test`, конечные точки в `backend/src/tests/`):
  - Модель `Test` включает обязательное поле `description` (64-2048 символов), которое используется как содержание теста в публичном банке.
  - Публичный банк тестов использует `/tests/public` и группирует опубликованные тесты по навыку, показывая только уровни, для которых есть хотя бы один опубликованный вариант.
  - Публичный ответ уровня теста включает `question_count`, `total_score`, `threshold_score`, `time_limit_minutes`, последний результат пользователя и доступность следующей попытки.
  - Прохождение теста запускается через `POST /tests/public/{skill_level_id}/start`: backend выбирает опубликованный вариант, который пользователь ещё не проходил, а если таких нет — наиболее давно пройденный.
  - Во время прохождения ответы отправляются через `POST /tests/attempts/{attempt_id}/answer`, завершение — через `POST /tests/attempts/{attempt_id}/finish`, а WebSocket `/tests/attempts/{attempt_id}/monitor` завершает попытку при внезапном разрыве соединения.
  - Ответы активной попытки не сохраняются в БД; в Redis хранится временное состояние до завершения, после чего создаётся или перезаписывается `TestAttempt` для пары пользователь-вариант теста.
  - Повторная попытка для уровня навыка доступна не раньше чем через `DAYS_FOR_TEST_ATTEMPT` дней после последней завершённой попытки по любому варианту этого уровня.
  - Страница прохождения теста находится на `/tests/attempt/:attemptId` и не отображает боковое меню.
  - Поиск тестов в разделе управления контентом поддерживает раздельные параметры: `keyword` для содержания и `skill_query` для навыка/уровня.
  - Количество вопросов в тесте — от 10 до 50.
  - Ограничение по времени в минутах — не менее 3.
  - Порог прохождения в баллах не превышает сумму баллов за все вопросы теста.
  - Балл за верный ответ на вопрос — не менее 1.
  - Длина описания вопроса — от 32 до 1024 символов.
  - Длина варианта ответа — до 64 символов.
- Для профессиональных стандартов:
  - `ProfStandard` хранит профессиональные стандарты серии 06 с уникальным числовым `code`.
  - `PsFunctionsGroup` хранит группы трудовых функций стандарта с уникальностью пары `ps_id` и `code`, а также уровень квалификации `qualification_level` от 1 до 9.
  - `PsFunction` хранит трудовые функции группы с уникальностью пары `ps_functions_group_id` и `code`.
  - API управления классификатором находится в `/classifier`: дерево возвращает `GET /classifier/tree`, CRUD ПС использует `/classifier/prof-standards`, CRUD ОТФ — `/classifier/groups`, CRUD ТФ — `/classifier/functions`.
  - Просмотр классификатора доступен ролям `curator` и `admin`; создание, редактирование и удаление ПС/ОТФ/ТФ доступно только роли `admin`.
- Для GitHub-интеграции:
  - Профиль GitHub хранится в `github_profiles`, где `id` соответствует идентификатору пользователя GitHub, а не локальному идентификатору.
  - Справочник репозиториев хранится в `github_repos`; поле `tokens` хранит последнюю оценку размера репозитория в токенах, а `user_repos` является связующей таблицей пользователя и репозитория с датой последнего анализа.
  - Выполнение практического задания через репозиторий фиксируется в `task_history`; сейчас успешное завершение анализа репозитория в рамках задания записывает `successful = true`.
- Для валидации учётных данных:
  - Имя пользователя содержит 4-16 символов и может состоять только из латиницы, кириллицы, `-` и `_`.
  - Поля ввода на страницах входа и регистрации ограничены 64 символами.
  - Пароль содержит 12-32 символа, минимум одну цифру, минимум один спецсимвол и минимум две буквы в разных регистрах.
- Для почтовых подтверждений:
  - Одноразовые коды подтверждения хранятся в Redis и передаются клиенту только внутри ссылок через параметр `code`.
  - Стандартная регистрация сначала подтверждает email через письмо, а аккаунт создаётся только после успешной проверки кода и заполнения оставшихся данных.
  - Регистрация через GitHub не требует дополнительного подтверждения email и использует существующий OAuth-поток.
  - Приглашения на роль куратора создаются администратором через `/management/curator-invitations` и хранятся в Redis один час; для нового email ссылка ведёт на форму завершения регистрации с ролью `curator`, для существующего пользователя ссылка подтверждает смену роли.
  - Раздел управления кураторами использует `/management/curators`; в списке отображаются пользователи с ролью `curator` или `admin`, а также активные Redis-приглашения. Для администраторов действие отзыва роли недоступно.
  - Отзыв роли куратора через `/management/curators/{user_id}/revoke` переводит пользователя на роль `user`, отзывает refresh-сессии и инвалидирует старые access-токены.
  - Смена почты подтверждается двумя письмами: первичный код отправляется на текущую почту пользователя, финальный код — на новый адрес.
  - Повторная отправка кода смены пароля или почты ограничена Redis-ключом на пользователя и доступна не чаще одного раза в минуту.
  - После успешной смены почты повторная смена блокируется Redis-ключом на пользователя на `DAYS_FOR_EMAIL_CHANGE` дней.
  - Успешные подтверждающие действия, меняющие учётные данные, должны отзывать все refresh-сессии пользователя и инвалидировать старые access-токены через Redis-маркер.
  - После глобальной инвалидации сессий backend отправляет SSE-событие `session_invalidated` в канал пользователя, а фронтенд очищает локальную сессию без ожидания следующего API-запроса.
- Для анализа репозиториев:
  - `backend/src/analysis/config.yaml` задаёт `limits.max_repository_payload_tokens`; при превышении лимита запрос к LLM не отправляется, а пользователю возвращается понятная ошибка через уведомление без численных лимитов.
  - Статус проверки репозитория «Подготовка» означает извлечение payload через gitingest и сохранение оценочного количества токенов перед отправкой в LLM.
- Для расчёта навыков пользователя:
  - Уверенность навыка рассчитывается по текущему интервалу уровня: максимум находится в середине интервала, а у границ текущего и следующего уровня множитель снижается.
  - Счётчик `obtained_count` для уровней навыков считается по достигнутому пользователем уровню: достижение третьего уровня засчитывается для первого, второго и третьего, а достижение первого — только для первого.
- Для вакансий (конечные точки в `backend/src/vacancies/`):
  - Запрос поиска вакансий поддерживает пагинацию через параметр `page` (начиная с 0).
  - Профессиональные роли для поиска вакансий берутся из `/professional_roles`, а роли с некоторыми идентификаторами и их дочерние узлы исключаются из IT-набора.
  - Подсказки регионов для фильтра вакансий берутся из suggest-метода HeadHunter и не кэшируются локально полным справочником.
  - Поле ключевых слов в поиске вакансий использует debounce-подсказки HeadHunter через отдельную конечную точку бэкенда.
  - Состояние фильтров, результатов и страницы поиска вакансий сохраняется на клиенте через Zustand-хранилище `useVacanciesStore`.

## Стиль кода

### Все языки
- Комментарии должны быть написаны только на русском языке, начинаться с маленькой буквы, без точки в конце комментария.

### Python
- Сигнатуры функций могут быть разбиты на несколько строк только в том случае, если функция имеет более трёх аргументов. Если аргументов три или меньше, оставляйте всю сигнатуру на одной строке.
- Если функция возвращает `None`, не аннотируйте тип возвращаемого значения — отсутствие аннотации возврата уже подразумевает `None`.
